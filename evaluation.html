Challenges and Solutions

  1. Integration with existing API
    -another API for this project was being created at the time where we joined in, meaning we would have to adapt and learn how the current one was being used. We had looked over it and discussed with the creator how it worked to understand it in its entirety.

  2. Integration with sister Team 31
    -for this project we had to work with Team 31 as the projects were related, this added a layer of complexity to both teams. To solve this we synchronised our work with Team 31 and kept them informed about our changed and when we made an important change that affected them or required their backend - we would work together to avoid conflict of work.

  3. Natural Language Processing
    -since the user would be at liberty to say whatever they wish to the bot or digital assistant it would be vital to find our what their intent would be from a variety of ways to express this. To overcome this complex task we used LUIS to train responses to give the correct intent from a given phrase.

  4. Team Management
    -the team had many separate tasks to work on but would also need to come together as a lot of different parts intertwined - this was solved by good team management and communication.

Critical Evaluation of project

General References
Area Critical Evaluation Description
Architecture design
UI and UX Since the final products were a chatbot and a digital assistant - this limited a lot of the options in terms of what we could focus on in order to improve the UI and UX. As a result we looked at what areas could be improved and really focused on making these the best they could be.

For the UI we had decided to use cards for the bots - this made the design much more visually appealing and made navigating the app more interesting yet also easier. The cards also allowed us to add functionalities such as booking in a much more convenient manner.

For the UX we spent a lot of time designing the user journey through the app and trying to get a user from the beginning of their conversation to the end of the conversation as quickly as possible. At the same time looked at how to make the transitions between reply and response as intuitive as possible so that a user would not be surprised or confused by a response to their query.
Functionality The bot and digital assistant both provide a high level of functionality as desired - they allow the users to book, view and cancel classes (and this can be easily expanded into other areas). They are both robust in the type and varied response of replies given back to the user.
Stability LUIS had recently changed for the MS Bot Framework which resulted in many changes having to be made to the natural language processing - however all parts of the project (including both the bot and digital assistant) function as epxected under reasonable queries and can stil deal with questions it would not expect.
Efficiency The aim was to take an intent from a user and then then get them to their desired action in as few steps as possible. This would be measured by number of messages and the number of utterances that would have to be made. As a lot of time has been spent reviewing user journeys - it has meant that as of now, the bot and digital assistant are both very good at handling queries and resolving them as fast as possible.
Compatibility The MS Bot platform allows the bots to be deployed over a range of bot platforms such as Facebook Messenger, Slack, Skype etc. This means it is very highly adaptable to whichever platform the client chooses to deploy the bot on.
Maintainability The main core of the app is contained within LUIS and Azure - anytime a change is wanted to be made and rolled out across all platforms, this can be changed in the respective MS portals and the changes will be automatically deployed across the various platforms. For more features to be added to the bot the Node code can be edited and tested - this makes it very easy to maintain the bot. The digital assistant is also centralised in Amazon's Developer site and as a result changes (e.g. a new feature) are very easy to implement.
Evaluation of Testing We had conducted two forms of test cases for this project, user testing and unit and integration testing.

User testing - as our bot would be used with a very wide audience, which could range from young teens to people in their old age, it was vital that the application would work for all demographics and would be as usable for one particular audience as well as another. For this we had made sure that the user would test all aspects of the app, from booking, viewing and cancelling a class to being able to go back and forth within an app with ease. These test had confirmed that our initial flow design and application designs had been fruitful as every user was able to use the application with ease. To keep this testing fair, we made sure that the demographic range of our users matches that of the end user audience.

Project management The project management was done well by the team leaders - information was always relayed to all members of the team and everyone had their own tasks to complete during each sprint. If help was required for any team members then they would be assisted accordingly. Furthermore, any new developments with the client or major changes would be communicated effectively.   • Ability to chat to a real-time assistant who may be working at the helpdesk for specialised requests

  • Ability to log into a user profile to view profile details
